# Kung Fu Chess — אפיון Client/Server

> מסמך זה הוא **אפיון בלבד** — שום דבר ממנו לא יְמומש עד שתשלחי מספר/שם איטרציה ספציפי מתוך הרשימה בסעיף 4. כל איטרציה תמומש בשלב נפרד, בצעדים קטנים, עם בקשת אישור לפני כל שינוי משמעותי (ובפרט לפני העברה/מחיקה של קבצים קיימים) — בהתאם לאופן העבודה שסוכם.

---

## 1. מהלך התכנון בקצרה

הפרויקט עובר מ"משחק מקומי אחד מול לוח" לארכיטקטורת **Client–Server**: השרת (בשלב הראשון — מקומי) מנהל משתמשים, דירוג, חדרים והתאמות, ומריץ את מנוע המשחק בפועל; הלקוח אחראי רק לקלט/פלט (קליקים, ציור) ולתקשורת עם השרת. השניים מדברים דרך **WebSocket**, והתקשורת הפנימית בצד השרת בין הרכיבים (Matchmaking, Rooms, Auth, Game Session) מתבצעת דרך **Event Bus** (Pub/Sub), כדי שהרכיבים לא יהיו תלויים ישירות זה בזה. חוקי המשחק, מודל הלוח וההתקדמות בזמן (`model` / `rules` / `realtime` הקיימים) עוברים ל־**`shared/`** כי הם נחוצים גם ללקוח (בדיקת חוקיות מקומית לפני שליחה לשרת) וגם לשרת (המקור היחיד לאמת). האפיון בנוי כסדרת איטרציות קטנות: תשתית תקשורת ריקה קודם, אחר כך Bus, אחר כך DB/Auth, אחר כך מסכי הבית/חדרים/matchmaking, ורק בסוף חיבור מנוע המשחק עצמו וה-UI הגרפי הקיים — כך שבכל שלב יש משהו רץ וניתן לבדיקה, והסיכון לקוד הקיים (`game-chess`) מרוכז בצעד ה-reorganization הראשון בלבד.

## 2. עקרונות ארכיטקטורה (תמצית)

- **הפרדה מלאה בין UI ללוגיקת משחק** — הלקוח לא מחליט מה חוקי; הוא רק בודק (מול `shared/rules`) לפני שליחה, וה־**שרת הוא הפוסק הסופי**.
- **Client / Server נפרדים לגמרי**, כל אחד בפרויקט/תיקייה משלו, מתקשרים רק דרך רשת (WebSocket), לא דרך imports ישירים אחד מהשני.
- **Shared** — הקוד היחיד שמותר גם ל-client וגם ל-server לייבא ישירות ממנו (לוח, מהלכים, חוקים).
- **Event Bus (Pub/Sub) בצד השרת** — Publishers מפרסמים אירוע ל-Bus; Subscribers/Listeners נרשמים לאירועים ספציפיים בלבד, בלי תלות ישירה ברכיב שפרסם.
- **FastAPI** בצד השרת — גם ל-WebSocket וגם (אם יידרש בהמשך) ל-endpoints רגילים.
- **SQLite** דרך `sqlite3`/thread ייעודי לכל בקשת DB (כדי לא לחסום את ה-event loop של FastAPI ב-I/O סינכרוני) — ללא GUI, ניהול ידני דרך Shell.
- **לוגים** משני הצדדים (client + server) על כל אירוע תקשורת/מערכת.
- **`engine_adapter` נשאר קבוע תחת `server/`** (לא עובר ל-`shared/`) — הוא זקוק לגישה ל-Bus ולמושגי Room/Session שהם server-only מטבעם; `shared/` נשאר "נקי" ממושגים כאלה כדי שגם לקוח שאין לו קשר לשרת יוכל תיאורטית להשתמש בו רק לבדיקת חוקיות מהלכים.
- **Matchmaking הוא Event-Driven, לא polling** — ראו פירוט מלא באיטרציה 6.
- **מבנה הודעות WebSocket** — envelope אחיד קבוע, ראו פירוט מלא באיטרציה 1.

## 4. איטרציות מימוש

כל איטרציה = יחידת עבודה עצמאית שאפשר לשלוח למימוש בנפרד. סדר האיטרציות נבחר כך שבסופה של כל אחת יש משהו שאפשר להריץ ולבדוק.

### איטרציה 0 — ריאורגניזציה של מבנה התיקיות
- יצירת `shared/`, `bus/`, `server/`, `client/` כתיקיות ריקות/עם שלד בלבד.
- העברת `game-chess/model`, `rules`, `realtime` → `shared/` (imports מתעדכנים בהתאם).
- העברת `game-chess/ui`, `input` → `client/` (`ui/`, `input/`).
- **כל העברה נעשית כצעד נפרד עם אישור**, ואחריה הרצת test suite הקיים (`pytest`) לוודא שכלום לא נשבר.
- שום פיצ'ר חדש לא נוסף בשלב הזה.

### איטרציה 1 — תשתית תקשורת בסיסית (שלד) + מבנה הודעות WebSocket
- `server/main.py` — FastAPI app + WebSocket endpoint ריק (echo בלבד).
- `client/network/` — חיבור WebSocket מהלקוח לשרת.
- `client/main.py` — הרצת CLI מינימלי שמתחבר לשרת, שולח הודעת טקסט, מקבל echo.
- לוגים בשני הצדדים על כל חיבור/הודעה.
- **מבנה הודעות (envelope) קבוע, זהה לשני הכיוונים**, נשלח כ-JSON דרך `websocket.receive_json()`/`send_json()` של FastAPI:
  ```json
  {
    "type": "string",       // סוג ההודעה, למשל "login" / "move" / "game_update" / "error"
    "payload": { ... },     // תוכן ספציפי לסוג
    "request_id": "string", // אופציונלי — לשיוך תגובה לבקשה ספציפית
    "ts": 0                 // אופציונלי — מילישניות, לצורך לוגים
  }
  ```
  - צד השרת מנתב לפי שדה `type` (dispatcher: `type → handler`); כל handler בפועל מפרסם אירוע מתאים ל-`bus/` או קורא ישירות ל-`room_manager`/`engine_adapter`.
  - סוגי הודעות **client→server** (רשימה ראשונית, תורחב בכל איטרציה רלוונטית): `login`, `register`, `play`, `cancel_play`, `create_room`, `join_room`, `cancel_room`, `move`, `jump`, `resign`.
  - סוגי הודעות **server→client**: `login_result`, `match_found`, `match_timeout`, `room_state`, `game_update`, `error`, `disconnect_countdown`, `rating_update`.
  - כל סוג הודעה חדש שנוסף באיטרציה מאוחרת יותר מתועד באותה איטרציה, לא כאן מראש.

### איטרציה 2 — Event Bus
- מימוש `bus/event_bus.py`, `bus/events.py`, שלד ל-`listeners/`, `publishers/`, `subscribers/`.
- אירוע לדוגמה אחד מקצה לקצה (למשל `ClientConnected`) — מתפרסם מ-`network/ws_routes.py`, נקלט ע"י listener שרושם ללוג — כדי לוודא שהמנגנון עובד לפני שבונים עליו לוגיקה אמיתית.

### איטרציה 3 — משתמשים, DB ו-Login
- `server/db/schema.sql` — טבלת `users` (username, password, rating — ברירת מחדל 1200).
- `server/db/users_repo.py` — קריאה/כתיבה, כל בקשה רצה ב-thread ייעודי (לא חוסמת את event loop).
- `server/auth/` — הרשמה/התחברות מול ה-DB.
- `client/cli/` — מסך Login (Command Line) ששולח פרטי משתמש לשרת ומקבל הצלחה/כישלון.

### איטרציה 4 — מסך הבית (שלד תפריט)
- לאחר Login מוצלח — תפריט CLI עם שתי אפשרויות: **Play** / **Room** (עדיין ללא לוגיקה — רק ניווט + שליחת הבחירה לשרת ואישור קבלה).

### איטרציה 5 — Room
- יצירת Room ID ייחודי (`Create`), הצטרפות לפי Room ID (`Join`), `Cancel`.
- הצגת ה-Room ID בראש המסך אצל כל המשתתפים בחדר.
- כלל: שני המשתתפים הראשונים = שחקנים, כל השאר = **Viewer**.
- אירועי Bus מתאימים (`RoomCreated`, `PlayerJoinedRoom`, `ViewerJoinedRoom`).

### איטרציה 6 — Matchmaking (Play) ו-ELO — Event-Driven
- **אין polling/סריקה מחזורית של תור ההמתנה.** ההתאמה מתבצעת אירועית:
  1. שחקן שולח `play` → השרת מוסיף אותו לרשימת ממתינים (`matchmaking.py`) ומפרסם `PlayerQueued` ל-`bus/`.
  2. Listener להתאמות מגיב **מיידית** לאירוע `PlayerQueued`: בודק אם יש כבר ממתין אחר בטווח דירוג ±100 ברשימה. אם כן — מפרסם `MatchFound` (נשלח כ-`match_found` לשני הצדדים) ומסיר את שניהם מהתור. אם לא — השחקן ממתין ברשימה עד שיגיע `PlayerQueued` הבא שמתאים לו, או עד timeout.
  3. ה-timeout של 60 שניות ממומש כ-callback מתוזמן (`asyncio` — למשל `loop.call_later`) שנוצר ברגע ה-enqueue; אם חלף בלי שהשחקן שובץ, מפרסם `MatchTimeout` (`match_timeout` ללקוח) ומסיר אותו מהתור.
  - היתרון: תגובה מיידית עם הצטרפות יריב מתאים, בלי latency של מרווח סריקה, ובלי צריכת CPU על scan תקופתי כשאין ממתינים.
- חישוב ועדכון ELO בסיום משחק (`server/logic/rating.py`).

### איטרציה 7 — חיבור מנוע המשחק בפועל
- `server/engine_adapter` (מיקום סופי, ראו סעיף 2) מפעיל את מנוע המשחק הקיים (`game-chess/engine` + `shared/rules`+`realtime`) בתוך `game_session.py`.
- מהלך: **הלקוח בודק תקינות מול `shared/rules` מקומית**; אם תקין — שולח לשרת; השרת מריץ את הבדיקה שוב (מקור אמת יחיד) ומשדר את התוצאה לשני השחקנים + לכל ה-Viewers בחדר.

### איטרציה 8 — טיפול בניתוקים ומקרי קצה
- ניתוק שחקן באמצע משחק → Auto-Resign אחרי 20 שניות, עם Countdown מוצג על המסך.
- מקרי קצה נוספים: הצטרפות משתמש נוסף לחדר קיים באמצע משחק (כ-Viewer), המתנה ממושכת ללא יריב (איטרציה 6), ניסיון הצטרפות ל-Room ID לא קיים.

### איטרציה 9 — חיבור ה-UI הגרפי הקיים
- העברת `client/ui` (renderer, sprite_manager, app-ui, **`img.py`** — ממשיכים להשתמש רק במחלקת `Img` הקיימת, ללא ספריית גרפיקה חדשה) לעבוד מול `client/network` במקום מול מנוע מקומי בלבד.
- קליקים מהממשק הגרפי → `client/input/controller` → בדיקה מקומית → שליחה לשרת → קבלת שידור-חזרה → ציור בפועל.

### איטרציה 10 (אופציונלי, לסוף) — ליטוש
- איחוד/סקירת לוגים, בדיקות (unit/integration) לרכיבי השרת, עדכון `Handoff.md`/מסמכי UI בהתאם למבנה החדש.

## 4א. איטרציות 11 ואילך — מסכי מעטפת גרפיים (Login / Home / Room / Matchmaking / Game Over)

**רקע ותלות:** איטרציה 9 חיברה את מסך המשחק הגרפי הקיים (`client/ui/renderer.py`, מבוסס `Img` בלבד) לשכבת הרשת — וזהו כבר המימוש בפועל היום: `client/main.py` מריץ thread רשת (asyncio, זרימת ה-CLI הטקסטואלי `cli/login.py → cli/home.py → cli/play.py|cli/room.py`) ו-thread ראשי שנשאר חסום עד שמתקבל אישות ש"המשחק התחיל", ואז פותח את חלון ה-`Renderer` — מסירה בין שני ה-threads דרך `client/network/game_bridge.py` (`GameBridge`, מבוסס `queue.Queue`). זהו הדפוס שאיטרציות 11–17 מרחיבות: **כל** מסכי המעטפת (לא רק תחילת משחק) עוברים לחלון גרפי יחיד באותו OS window, עם אותה חלוקת thread (רשת אסינכרונית מול GUI סינכרוני מבוסס `cv2`/`Img`).

**מה נשאר בדיוק כפי שהוא, בלי שום נגיעה, בכל האיטרציות האלה (למעט חריג אחד ומאושר מראש באיטרציה 16):**
- `client/ui/renderer.py`, `client/ui/sprite_manager.py`, `client/ui/img.py` — מסך המשחק עצמו, כולל כל האנימציות (תנועה חלקה, קפיצות, פסי מנוחה) נשאר פיקסל-לפיקסל זהה למה שאיטרציה 9 כבר מימשה.
- `client/network/connection.py`, `client/network/game_bridge.py` — נשארים כפי שהם; `GameBridge` ממשיך לשרת רק את מסירת "המשחק התחיל" בדיוק כפי שהיום.
- `client/cli/*.py` (`login.py`, `home.py`, `room.py`, `play.py`) — **לא נמחקים ולא נערכים**. הם ממשיכים לשמש את `client/text_test/script_runner.py` (תרחישי בדיקה אוטומטיים דרך קלט טקסטואלי). מסכי המעטפת הגרפיים הם קוד **נוסף**, לא תחליף שנוגע בקוד הקיים.
- כל שכבת `model`/`rules`/`realtime`/`engine_adapter` — אין שום נגיעה; האיטרציות האלה הן UI+bridging בלבד.

**עקרון DRY מחייב לכל האיטרציות שלהלן:** רכיב גנרי נבנה **פעם אחת** באיטרציה 11 (`widgets.py`, `screen_manager.py`, `app_bridge.py`, `base_screen.py`) ונעשה בו שימוש חוזר מכל מסך; אין שכפול לוגיקת כפתור/שדה-קלט/חיבור-רשת בין המסכים השונים.

### איטרציה 11 — תשתית משותפת למסכים גרפיים (Screens Framework)
**תלות:** איטרציה 9 (משתמשת ב-`WINDOW_NAME` הקיים של `renderer.py` — ראו למטה — כדי שכל המסכים, כולל מסך המשחק, יחיו באותו חלון OS יחיד).
- `client/ui/widgets.py` — רכיבי UI גנריים מבוססי `Img`/`cv2` בלבד (ללא שום ספריית גרפיקה נוספת, כמו הכלל הקיים ב-`renderer.py`):
  - `Button` — מלבן + טקסט ממורכז + בדיקת "פגיעה" (hit-test) לפי קואורדינטת קליק.
  - `TextInput` — שדה טקסט עם מצב פוקוס, מצב מוסתר (`•••`) לסיסמאות, טיפול ב-Enter/Backspace — לפי אותו דפוס מקלדת שכבר קיים ומוכח ב-`Renderer._prompt_text` (לא מועתק code-for-code; ממומש פעם אחת כאן ומשמש את כל שדות הטקסט העתידיים).
  - `Label`/`ErrorText` — טקסט סטטי/הודעת שגיאה בצבע ייעודי.
- `client/ui/screens/base_screen.py` — מחלקת בסיס `Screen`: `render(canvas: Img)`, `handle_click(x, y)`, `handle_key(key)`, `on_enter(payload)` (נקרא כשנכנסים למסך, מקבל payload מהמסך הקודם), `update()` (נקרא כל frame; בודק אירועים חדשים מה-bridge). מסך מסמן מעבר הלאה ע"י הצבת `self.next_screen = (ScreenClass, payload)`.
- `client/network/app_bridge.py` — `AppBridge`, הכללה של `GameBridge` הקיים (לא מחליף אותו — קובץ נפרד, נוסף):
  - `send_request(envelope)` — נקרא מה-thread הראשי; מעביר בקשה ל-thread הרשת דרך `asyncio.run_coroutine_threadsafe` (כמו שכבר עושה `game_bridge.build_remote_engine.send_move` היום), ושולח בפועל דרך אותו `ServerConnection` הקיים.
  - לולאת קבלה רציפה אחת בצד ה-thread הרשת (מחליפה את הקריאות הנקודתיות ל-`connection.receive()` שמפוזרות היום בכל אחד מקבצי `cli/*.py` — אך **לא** נוגעת בקבצי ה-CLI עצמם, שממשיכים להשתמש ב-`connection.receive()` הישיר כרגיל בזרימה הטקסטואלית שלהם) — מסווגת כל מעטפת נכנסת: תגובה ישירה לבקשה אחרונה, או broadcast לא-מבוקש (כמו `room_state`, `match_found`) שמצטבר בתור אירועים למסך הפעיל.
  - `poll_events()` — נקרא לא-חוסם מה-thread הראשי בכל frame (מתוך `update()` של המסך הפעיל), מחזיר את כל האירועים שהצטברו מאז ה-poll הקודם.
  - כמו ב-`GameBridge`, שגיאת חיבור/ניתוק (`websockets` exception) מתורגמת לאירוע פנימי `connection_lost` שמצטבר באותו תור אירועים — כל מסך מטפל בו לפי ההקשר שלו (ראו איטרציות 12–15).
- `client/ui/screen_manager.py` — הלולאה הגרפית הראשית שרצה על ה-thread הראשי: פותחת חלון `cv2` יחיד (מייבאת את `WINDOW_NAME` מ-`client.ui.renderer` — **ייבוא קריאה בלבד, שום עריכה של הקובץ** — כדי להבטיח שגם מסך המשחק, כשהוא נפתח בהמשך הזרימה, ישתמש באותו שם חלון פיזי ולא יפתח חלון שני), רושמת `cv2.setMouseCallback` אחד שמנתב קליקים למסך הפעיל, ורצה: `update()` → `render()` → `imshow`/`waitKey` → ניתוב קליק/מקש → אם `next_screen` הוגדר, מחליפה מסך (קוראת ל-`on_enter` של החדש עם ה-payload).
- **קריטריון קבלה לאיטרציה זו:** שני מסכי-דמה מינימליים (בלי שום קריאת רשת אמיתית) עם כפתור אחד שעובר מהראשון לשני ומעביר payload — מוכיח שהמנגנון (חלון, widgets, bridge, מעברים) עובד, לפני שבונים עליו את המסכים האמיתיים.

### איטרציה 12 — מסך Login גרפי, ו-`client/main.py` עובר לזרימה הגרפית כברירת מחדל
**תלות:** איטרציה 11.
- `client/ui/screens/login_screen.py` — כותרת, שדות `TextInput` ל-username ול-password (מוסתר), כפתור Login/Register (מתג בין השניים), שורת שגיאה, מצב "מתחבר..." בזמן המתנה לתשובת שרת.
- זרימה: בלחיצה על שליחה — **תחילה בדיקת תקינות בצד לקוח** (שדות לא ריקים; שום קריאת רשת על קלט ריק — עקרון "הלקוח בודק לפני שליחה" שכבר נקבע לגבי מהלכים באיטרציה 7, מיושם כאן גם לטופס): אם תקין, `AppBridge.send_request(Envelope(type="login"/"register", ...))`; בקבלת תגובה — הצלחה → מעבר ל-`HomeScreen` עם ה-username/rating כ-payload; כישלון → הודעת השרת מוצגת inline, המשתמש נשאר על אותו מסך ויכול לנסות שוב בלי להפעיל את התהליך מחדש.
- **מקרי קצה מוגדרים כעת בתכנון (לא "יטופלו" אד-הוק במימוש):**
  - שדה ריק → נחסם בצד לקוח, אין קריאת רשת.
  - כשל התחברות לשרת (`ServerConnection.connect()` נכשל לפני שהחלון אפילו עולה) → מסך מציג הודעת שגיאה קבועה + כפתור "נסה שוב" שמפעיל ניסיון חיבור נוסף.
  - ניתוק באמצע המתנה לתשובת login/register (`connection_lost` מה-bridge) → אותה הודעת שגיאה + "נסה שוב", בלי לסגור את האפליקציה.
  - פרטים שגויים / שם משתמש תפוס בהרשמה → משתמש באותו חוזה `success:false` + `message` שכבר קיים היום ב-`login.py`, רק מוצג גרפית במקום ב-`print`.
- **נגיעה מפורשת ומאושרת מראש בקובץ קיים:** `client/main.py` — הזרימה שמריצה היום `do_login` + `run_home_menu` (טקסטואלי) מוחלפת ב-`AppBridge` + `ScreenManager(start=LoginScreen)`. `client/cli/*.py` עצמם לא נערכים (ראו הערת התלות למעלה) — הם פשוט מפסיקים להיות נקראים מ-`main.py` (אך ממשיכים לשמש את `script_runner.py`).
- **קריטריון קבלה:** הרצת `python -m client.main` פותחת ישירות את מסך ה-Login הגרפי; login תקין עובר למסך Home (עדיין placeholder בשלב זה); login שגוי מציג שגיאה inline ומאפשר ניסיון חוזר.

### איטרציה 13 — מסך Home (תפריט) גרפי
**תלות:** איטרציה 12.
- `client/ui/screens/home_screen.py` — מציג את שם המשתמש המחובר + הדירוג (מגיע כ-payload מ-`LoginScreen`), שני כפתורים: **Play** (שולח `menu_select`/`play`, עובר ל-`MatchmakingScreen`) ו-**Room** (שולח `menu_select`/`room`, עובר ל-`RoomScreen`) — תואם ל-2 האפשרויות הקיימות ב-`cli/home.py`, בלי הוספת אפשרויות שלא התבקשו (כמו Logout).
- **מקרה קצה:** ניתוק בזמן שהיית המתנה סבילה על מסך הבית (`connection_lost` מה-bridge) → הודעת שגיאה + כפתור "חזרה למסך Login" (לא ניסיון-חיבור-מחדש שקט — משתמש צריך להתחבר מחדש).
- יציאה מהאפליקציה נשארת דרך סגירת החלון/מקש quit, באותו מוסכם שכבר קיים ב-`Renderer.render()`.
- **קריטריון קבלה:** אחרי login מוצלח, לחיצה על Play או Room שולחת בפועל את ההודעה המתאימה לשרת ועוברת למסך הבא (עדיין placeholder לפני איטרציות 14–15).

### איטרציה 14 — מסך Room גרפי (Create / Join / המתנה)
**תלות:** איטרציה 13. מאגד Create+Join+מסך-המתנה כיחידה אחת (תואם למבנה הקיים ב-`cli/room.py` — `run_room_menu` + `_wait_in_room` כפונקציה אחת רציפה).
- `client/ui/screens/room_screen.py` — תת-מצבים על אותו מסך:
  1. **בחירה:** כפתורי Create / Join.
  2. **Join:** שדה `TextInput` ל-Room ID + כפתור אישור.
  3. **המתנה:** Room ID מוצג בבירור בראש המסך (כנדרש כבר בסעיף 3/איטרציה 5 של התכנון), עדכון מספר משתתפים/צופים לפי הודעות `room_state` שמגיעות מה-bridge, כפתור **Cancel**.
- אירועי bridge: `room_state` → עדכון תצוגת ההמתנה; `game_started` (מגיע רק לשני השחקנים, בדיוק כמו היום) → מעבר **לא** למסך גרפי חדש אלא ישירות לזרימת המשחק הקיימת מאיטרציה 9 (`build_remote_engine` + `Renderer`) — שימוש חוזר מלא, ללא שכפול.
- **הבהרת scope, לא הרחבה:** צופים (viewers) שמצטרפים לחדר באמצע משחק פעיל נשארים, כמו היום ב-`cli/room.py` (ההערה בקוד "viewers stay on the CLI room screen for this iteration"), על מסך ההמתנה הגרפי במצב "צופה" — בלי ציור לוח-צפייה בפועל; זו אינה נסיגה אלא המשך אותה מגבלה קיימת ומתועדת. אם בעתיד יידרש מסך-צפייה גרפי אמיתי, זו נקודה פתוחה נפרדת (ראו סעיף 5).
- **מקרי קצה:** הצטרפות ל-Room ID לא קיים → `success:false` מהשרת מוצג inline על תת-מצב ה-Join, בלי לצאת מהמסך; Cancel מתוך מסך ההמתנה → שולח `cancel_room`, חוזר ל-`HomeScreen`; ניתוק (`connection_lost`) בכל תת-מצב → הודעת שגיאה + חזרה ל-Login (כמו איטרציה 13).
- **קריטריון קבלה:** Create מציג Room ID ומחכה; פתיחת חיבור שני שמצטרף לאותו Room ID (Join) מעבירה את שני הצדדים למסך המשחק הגרפי הקיים בלי שום מסך ביניים טקסטואלי.

### איטרציה 15 — מסך Matchmaking (Play) גרפי
**תלות:** איטרציה 13.
- `client/ui/screens/matchmaking_screen.py` — "מחפש יריב..." + זמן שחלף (עד 60 שניות, לפי מנגנון ה-timeout הקיים מאיטרציה 6), כפתור **Cancel**.
- אירועי bridge: `match_found` → מוצג שם/דירוג היריב inline, נשאר על אותו מסך (עדיין לא game_started בפועל); `game_started` → מעבר ישיר לזרימת המשחק הקיימת (כמו באיטרציה 14, אותו שימוש חוזר); `match_timeout` → הודעת "לא נמצא יריב" + כפתור חזרה ל-Home (תואם להודעה הקיימת ב-`cli/play.py`).
- **מקרה קצה מפורש (race שכבר קיים היום בלוגיקת ה-CLI וצריך להתקיים גם כאן):** אם `game_started` מגיע מהשרת אחרי שהמשתמש כבר לחץ Cancel (ה-`cancel_play` כבר בדרך או כבר נשלח) — המסך חייב עדיין לפתוח את חלון המשחק כשה-`game_started` מגיע, בדיוק כמו שהדגל `resolved` ב-`cli/play.py:_wait_for_match` כבר עושה זאת היום; אותה עדיפות-לוגית מיושמת כאן דרך סדר בדיקת אירועי ה-bridge (בדיקת `game_started`/`match_timeout` שהתקבלו לפני שמפעילים בפועל את הודעת ה-Cancel).
- **קריטריון קבלה:** לחיצה על Play במסך הבית פותחת את מסך ה-matchmaking; חיבור שני שמצטרף לתור באותו טווח דירוג מעביר את שניהם למסך המשחק הגרפי הקיים.

### איטרציה 16 — מסך Game Over גרפי + חזרה לתפריט
**תלות:** איטרציה 9 (`renderer.py`) ואיטרציה 11 (`widgets.py`).
- **הכרעת עיצוב שנבחרה:** לא נבנה מסך נפרד אחרי סגירת חלון המשחק — במקום זאת, כפתור "חזרה לתפריט" (רכיב `Button` הקיים מאיטרציה 11) מצויר **בתוך** אותו חלון משחק קיים, ברגע ש-`game_engine.is_over` הופך ל-`True`, ממש לצד/מתחת לטקסט "GAME OVER" הקיים.
- **הנגיעה היחידה, המפורשת והמאושרת-מראש בקוד קיים ועובד בכל תכנון האיטרציות 11–17 (`client/ui/renderer.py` ו/או `client/main.py`):**
  - ב-`renderer.py`: כשה-loop מזהה `is_over`, מציירים בנוסף (לא במקום) כפתור אחד קבוע-מיקום מעל/ליד טקסט ה-GAME OVER הקיים; קליק על שטח הכפתור מוחזר כסימן החזרה-לתפריט מובחן מ"סגירת חלון רגילה" (שממשיכה להיות quit מלא כמו היום).
  - ב-`client/main.py` (או המקבילה שלו ב-12): הקורא ל-loop של המשחק (`_run_graphical_game` כיום) מטפל בערך ההחזרה החדש הזה ע"י מסירת שליטה בחזרה ל-`ScreenManager` שפותח `HomeScreen` מחדש (באמצעות אותו `AppBridge`/`ServerConnection` — **בלי** login מחדש), במקום לסיים את התהליך.
  - שום דבר אחר ב-`renderer.py` (ציור לוח, כלים, אנימציות, פאנלים, פסי מנוחה, שמות שחקנים) לא משתנה.
  - **לפני מימוש בפועל של איטרציה זו — ה-diff המדויק ב-`renderer.py`/`main.py` יוצג לאישור נקודתי**, כנדרש בכללי העבודה (נגיעה בקובץ קיים תמיד באישור מראש, גם כשההכרעה העקרונית כבר סוכמה כאן).
- **קריטריון קבלה:** סיום משחק (מט/פרישה) מציג את אותו מסך GAME OVER כמו היום, עם כפתור נוסף שמחזיר בפועל למסך הבית הגרפי בלי לסגור את התהליך ובלי מעבר דרך טרמינל.

### איטרציה 17 — ליטוש וכיסוי קצה-לקצה של הזרימה הגרפית
**תלות:** איטרציות 11–16 (וכן 8, 9, 10 הקיימות).
- מעבר ידני מלא על כל המסלול: Login → Home → (Play/Matchmaking **או** Room) → משחק פעיל (זהה ל-UX שהיה קיים לפני שילוב client/server) → Game Over → Home — ומוודאים שאין אף נקודה שמחזירה את המשתמש לטרמינל טקסטואלי.
- בדיקה שה-Countdown לניתוק/Auto-Resign (איטרציה 8) ממשיך להיות מצויר נכון בתוך מסך המשחק הבלתי-משתנה גם כשמגיעים אליו דרך נתיב הכניסה הגרפי החדש.
- ריצת רגרסיה שמוודאת ש-`client/text_test/script_runner.py` (התרחישים הטקסטואליים דרך `client/cli/*.py`) ממשיך לעבוד ללא שינוי — מוכיח ששני הנתיבים (CLI ל-scripted tests, גרפי כברירת מחדל) חיים זה לצד זה בלי לפגוע אחד בשני.
- עדכון תיעוד (`Handoff.md`/מסמכי UI אם קיימים) לזרימת המסכים החדשה — תיעוד בלבד, בלי שינוי התנהגות.

## 5. מה עדיין פתוח / יוחלט בזמן המימוש בפועל

שלוש ההחלטות שהיו פתוחות כאן (מיקום `engine_adapter`, מנגנון matchmaking, מבנה הודעות WebSocket) **הוכרעו** — ראו סעיף 2 ואיטרציות 1, 6, 7 בהתאמה.

לגבי איטרציות 11–17 (סעיף 4א) — שלוש הכרעות ארכיטקטורה שהיו פתוחות **הוכרעו** מראש, לפני כתיבת האיטרציות:
- דפוס ה-threading למסכי המעטפת הגרפיים: הרחבת דפוס `GameBridge` הקיים (`AppBridge` חדש ונפרד) — לא שינוי בלולאת האסינכרוניות הקיימת. ראו איטרציה 11.
- מסך Game Over: כפתור מוטמע בתוך חלון המשחק הקיים (נגיעה נקודתית ומאושרת-מראש ב-`renderer.py`/`main.py`), לא מסך נפרד. ראו איטרציה 16.
- נקודת הכניסה: `client/main.py` עצמו עובר לזרימה הגרפית כברירת מחדל; `client/cli/*.py` נשארים ללא שינוי לשימוש `script_runner.py`. ראו איטרציה 12.

נקודה פתוחה חדשה אחת, שלא נדרשה כרגע ולכן לא נכנסה לסקופ של איטרציה 14: **מסך צפייה גרפי לצופים (viewers)** שמצטרפים לחדר באמצע משחק — כרגע (גם ב-CLI הקיים וגם באיטרציה 14 המתוכננת) צופה נשאר על מסך ההמתנה במצב "צופה" בלי ציור לוח בזמן אמת. אם יידרש בעתיד — זו איטרציה נפרדת שתוסיף אחריה למספור (18 ואילך), ותצטרך להכריע אם היא בונה על `Renderer` הקיים במצב read-only או רכיב נפרד.

אין כרגע נקודות פתוחות נוספות; אם יתגלה צורך בהחלטה נוספת במהלך מימוש איטרציה ספציפית, היא תוצג לאישור באותה נקודה.
